# 为什么 `zero_byte_1` 反而比 `zero_byte_2` 慢？

你的直觉（无分支 + ILP 应该更快）在**通用情况下**是对的，但这里有几个关键因素让结论翻转了。

## 为什么"看起来更并发"的版本反而更慢？

### (a) 总指令数：17 vs 平均 ~10

- `zero_byte_1`：固定 ~17 条指令，**不管输入是什么都得全跑**。
- `zero_byte_2`：序言 5 条 + 循环体 7×N 条，N 是 zero byte 出现的最高字节位置（1~4，或 0 直接退出）。
    - 输入 `0x01020300`（zero byte 在第 1 字节，N=1）：5 + 7×1 = **12 条**

### (b) 串行依赖链长度（关键！） (只分析存在差异的代码片段，分析的有点不太对)

`zero_byte_1` 中所有 `shr/and/or` 都串在 `ret1 |= ret2 |= ret3 |= ret4` 这条**或链**上：

```
edi → shr7 → and1 →  or →    or →   or → eax(ret)
      edx    edx      ↑      ↑      ↑
                 shr14&3   shr21&7  shr28
                  %eax     %edi    %edx                 
```

最后 3 个 `or` 是**串行**的（`or %edx,%eax; or %edi,%eax`），形成一条长度为 5~6 的依赖链。CPU 必须等这条链跑完才能 ret，下一次 `zero_byte_1` 调用又要从头开始（虽然不同调用之间可以乱序，但每次调用内部链很长）。

`zero_byte_2` 循环体里的依赖：

```
edx → shr7 → and1 → or → eax(ret)
                    ↑
                add eax
edx → shr8 → edx (下一轮)
```

每轮循环只有 **2~3 条依赖链长度**，并且 `add %eax,%eax`（即 `ret2<<1`）和 `shr %ecx` 可以在不同执行端口并行。每轮关键路径只有 ~3 cycle。

虽然循环总指令数可能更多，但**关键路径短**，CPU 端口利用率更高。

### (c) 分支预测几乎 0 开销

10 亿次同一个输入 → 同一个循环次数 → 分支预测命中率 ~100%，`jne` 实际上是免费的。这就抹掉了"无分支"原本的优势。

### (d) 微妙的瓶颈：`zero_byte_1` 端口压力集中在 shifter

x86 上 `shr` 通常只能走 1~2 个端口（如 Intel 的 P0/P6）。`zero_byte_1` 一次性发 4 个 `shr`，全部争抢 shifter 端口；而 `zero_byte_2` 的 shr 分散在循环不同迭代里，**反而错峰执行**，IPC 更高。

##  综合结论

| 维度 | `zero_byte_1`（无分支） | `zero_byte_2`（循环）   |
|---|---|---------------------|
| 总指令数 | 固定 17 | ~5 + 7×N（N=1 时约 12） |
| 关键路径长度 | **长**（4 路 OR 串成链） | **短**（每轮 ~3 cycle）  |
| Shifter 端口争用 | 严重（4 个 shr 集中） | 轻（分散在迭代）            |
| 分支开销 | 0 | ~0（预测 100% 命中）      |
| 输入相关 | 无 | 有早退优势               |

**真正的瓶颈不是"分支 vs 无分支"，而是"关键路径长度 + 端口压力"。** 在这个特定输入和现代乱序 CPU 上：

- 无分支并不等于 ILP 高，需要看依赖链能不能真正并行。
- `zero_byte_1` 表面上 4 个子表达式独立，但最后用 `|` 串起来 → 实际形成长依赖链，并且 shr 端口拥塞。
- `zero_byte_2` 的循环在分支预测器面前几乎免费，每轮关键路径短，整体反而更快。

## 一句话总结

> **"无分支"≠"ILP 好"**。`zero_byte_1` 看似 4 路并行，实际被尾部串行 OR 链 + shifter 端口拥塞拖累；`zero_byte_2` 的循环在完美分支预测下几乎免费，且关键路径更短，所以反而更快。要让 `zero_byte_1` 真正赢，得缩短关键路径（如树形 OR、`imul` 收集 sign bit、或 `pext`）。
